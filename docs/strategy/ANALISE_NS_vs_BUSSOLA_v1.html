Mensagem central

Navegar a documentação pública do NS valida o rumo da Bússola em praticamente todos os pontos estratégicos. Nenhuma frente P0/P1 precisa mudar. Surgiram 4 padrões concretos que o SSE pode adotar sem violar o posicionamento "simpler + cheaper + purpose-built" — e 13 decisões de descope ganham base objetiva para encerrar debate recorrente.

6 achados em 30 segundos

1

Centers = Workspaces

NS usa "Centers" por role. É literalmente o que RF-001 propõe. Nosso padrão é industry-standard, não experimento — reduz risco de adoção.

2

FSM Mobile valida Gap 2

NS entrega field service mobile nativo com offline. É referência concreta que o mercado conhece. Sem Gap 2, SSE não compete em mobility.

3

Customer 360 é o ganho mais claro

Tela unificada com abas (Overview, Vehicles, Estimates, SOs, Receivables, Activity). Substitui navegação fragmentada. Proposto como RF-004.

4

Payment Hold + Available Balance

Conceitos financeiros do NS que encaixam no SSE: estimate disputado bloqueia SO (RF-006); KPI "Cash disponível" do Cockpit distingue saldo real vs available.

5

Onde SSE supera NS

1099-NEC nativo (NS usa Yearli/Sovos externos), MACRS nativo, onboarding wizard <30min, activation tracking instrumentado. Reforçar narrativa comercial.

6

Descopes deliberados explícitos

4 áreas onde NS tem features avançadas que NÃO entram no SSE: SuiteBilling, OneWorld/Multi-subsidiary, SuiteFlow, Dashboard drag-and-drop. Documentado = menos debate.

⚙️

Ajuste no RF futuro do Cockpit (Gap 4)

KPI "Cash disponível" deve distinguir Cash Balance vs Available Balance — padrão copiado de NS In-Transit Payments.

Exibição: "$XX,XXX available ($YY,YYY total)"

Por quê: sem essa distinção, Owner vê $50k no sábado, aprova gasto de $30k, e na terça descobre que $25k estava comprometido em vendor bills em trânsito. É a exata falha que Bússola §3 descreve como "dashboard atual é operacional, não gerencial".

§5 — Onde simplificamos (expansão)

Adiciona 6 linhas à tabela, explicitando descopes hoje implícitos.

+ Custom Segments / Classifications — NS: Dept/Class/Location/CS | SSE: tenant = 1 shop = 1 location. Violação direta de P7.
+ SuiteFlow / Workflow Designer — NS: BPMN visual builder | SSE: n8n workflows pré-definidos.
+ Saved Searches / Custom KPIs — NS: user monta query → vira KPI | SSE: 4 relatórios canônicos + KPIs fixos por persona.
+ OneWorld / Subsidiaries — NS: multi-entity | SSE: 1 tenant = 1 empresa. Anti-target §1.
+ SuiteBilling / Usage billing — NS: recurring + usage + overage | SSE: Stripe plano fixo. Fase 7+.
+ Payment Holds genérico — NS: flag em qualquer bill | SSE: específico para supplements disputed (RF-006).

§5 — Onde superamos (reclassificação)

Promover 1099-NEC + adicionar MACRS nativo.

~ 1099-NEC nativo sai de "Herdamos" → entra em "Superamos". NS não gera 1099; depende de integradores (Yearli/Sovos/Track1099).
+ MACRS nativo — NS trata como "alternate tax method". SSE tem nativo no FAM (ADR-008) porque é requisito IRS para veículos do shop.

§6 — Novo princípio P8

Offline-first para shop floor — derivado de FSM Mobile offline.

+
P8 — Offline-first para shop floor.

Técnico não pode perder trabalho quando WiFi cai. Operações críticas do mobile (timer, fotos, SO status) funcionam offline e sincronizam quando reconectar. Desktop pode assumir online; mobile do técnico não.

Racional: gap concreto identificado em FSM Mobile. Body shop tem WiFi ruim tipicamente. Sem offline, mobile vira "ferramenta que só funciona no escritório" — contradiz P2.

§7 — Navegação (notas adicionais)

+ Nomenclatura: adotamos "Workspace" (não "Center" como NS) — autodescritivo em EN/PT-BR, não importa jargão NS.
+ Global Search (Cmd+K) é obrigatório em todos os workspaces. Se não existir no SSE, é ENH P1 independente.

ADR-012 — Incorporação parcial de padrões NS à Bússola v1.1

Status: ACCEPTED Data: 2026-04-21 Autor: Luigi (PO) + PO Assistant

Contexto

Em 2026-04-21, sessão PO Cowork realizou análise comparativa entre a documentação pública do NS e a Bússola de Produto SSE v1.0 (ADR-009). Análise documentada em docs/strategy/ANALISE_NS_vs_BUSSOLA_v1.md.

Opções consideradas

Opção Risco Custo Reversão
A. Adotar ajustes Baixo — adições, não redesenho 1 sessão PO Revisão trimestral Julho/2026
B. Esperar revisão trimestral Perda de janela 2–3 meses Zero N/A
C. Descartar aprendizados Contradiz evolução Bússola Zero Histórico como insumo

Decisão recomendada

Opção A — Adotar ajustes.

Baixo risco + janela aberta para incorporar em Fase 2 + P8 endereça gap concreto do Gap 2.

Condição de reversão

  • • Se ≥2 dos novos RFs (004–007) forem reavaliados como overhead e cancelados antes de dev → reabrir este ADR.
  • • Se P8 atrasar Gap 2 em >30 dias → reavaliar P8 como "nice-to-have, não requisito".

Consequências

POSITIVAS
  • • Bússola v1.1 com 13 anti-recs explícitas
  • • 4 RFs adicionados com escopo claro
  • • P8 alinha Gap 2 com padrão indústria
  • • §5 mais robusta
NEGATIVAS
  • • Custo sessão PO para formalizar v1.1
  • • Pressão no Fase 2 (RF-004 + RF-006 P1)
  • • Redação de 4 novos RFs
